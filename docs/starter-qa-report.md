# AuthToolkit Identity Starter QA Report

Date: 2026-06-27

## Result

Starter QA passed for the local starter shell. The app runs locally, core routes load safely, docs are present, and no raw Identity server secrets were exposed during QA.

## Local target

- Local app: `http://localhost:3071`
- Command used: `PORT=3071 npm run dev`
- Deployment: not run

## Security checks

- Server-only env names are kept out of browser pages.
- `.env.local` is ignored.
- Session cookie is HTTP-only.
- Starter session is signed with `AUTHTOOLKIT_IDENTITY_SESSION_SECRET`.
- Callback route does not dump raw upstream Identity responses.
- Protected `/app` checks the starter session before rendering.
- No raw secrets were printed in QA output or committed.

## Issues found

- Dependencies were not installed before QA. Running `npm install` created `package-lock.json`, which should stay committed for repeatable starter installs.
- Next dev updated TypeScript setup for local dev route types. The generated `next-env.d.ts` change was normalized back to starter-friendly content; `tsconfig.json` keeps the `.next/dev/types/**/*.ts` include so local dev remains compatible.
- The verified callback path requires `AUTHTOOLKIT_IDENTITY_SESSION_SECRET`. Without it, the page fails safely. This is expected until a developer creates `.env.local`.

## Fixes made

- Added this QA report.
- Added `package-lock.json` from `npm install` for repeatable dependency installation.
- Kept the local dev TypeScript include generated by Next.

## Remaining pending items

- Real hosted Identity sign-in requires real Identity project credentials in `.env.local`.
- Callback URL must be added to the Identity allowed origins before a real local handoff.
- Production deployment was not tested in this QA pass.

## Readiness decision

The starter is ready for beginner local setup and real Identity credential testing. It is not production-validated until a developer configures real env values, runs the hosted handoff, and deploys to a server-capable host.
